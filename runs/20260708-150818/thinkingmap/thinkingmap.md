# Thinking Map: Battery Impedance Meter System

## project_summary
基于 STM32G030F6P6，通过 1kHz 交流恒流注入与 10kHz 同步采样，实现单点电池内阻 (mΩ 级) 精确测量，并作为级联网络中的智能节点，完成数据采集、处理与上报。

## system_archetype
`heterogeneous_rtos_bare_metal` — 这是一个**裸机+状态机**系统，但通过定时器中断和 DMA 实现了准实时多任务调度，本质上是一个微内核的任务调度器。

## angle_sweep
### 【1. 技术栈与关键路径 (Technology & Critical Path)】
*   **MCU/SoC/RTOS**: STM32G030F6P6 (Cortex-M0+，无硬件除法和浮点单元，依赖CMSIS-DSP软件库)。
*   **编译器工具链**: ArmClang V6.19 (via Keil MDK-ARM v5)。 **注意：ArmClang 对 `volatile` 和内存对齐的优化行为与 GCC 不同，需特别关注。**
*   **关键时序路径 (Critical Timing Path)**:
    1.  **ADC 采样链路**: `TIM3 更新事件 (10kHz)` -> `ADC1 注入组启动` -> `ADC 采样保持 (若干 ADC 时钟周期)` -> `ADC 转换完成` -> `ADC1_IRQHandler` 进入。
    2.  **数据计算链路**: `ADC1_IRQHandler` 内 `DMA 读取` -> `软件相位补偿` -> `内阻计算 (CMSIS-DSP)` -> `计算结果存入全局变量`。
    3.  **DS18B20 时序链路**: `TIM14 定时器中断 (例如 5us)` -> `软件模拟 OneWire 时序` -> `读取温度` -> `数据校验`。
*   **核心算法**:
    *   **内阻计算**: **同步解调 (Synchronous Demodulation)**: 将采集到的电压、电流信号与 1kHz 参考信号 (由 TIM3 计数产生) 做相关运算，提取基波幅值和相位，从而计算出电阻 (R = V/I * cos(θ))。这是抵抗噪声的最有效方法。
    *   **相位校准**: 使用已知纯电阻 (如 0.1Ω 采样电阻) 进行标定，通过测量得到的复数阻抗的虚部来推算系统固有的相位延迟，然后在后续计算中进行补偿。
    *   **异常数据识别**: **MAD (Median Absolute Deviation)** 方式：维护一个长度为 N (如 N=5) 的内阻测量值滑动窗口，若最新值与窗口中位数的偏差超过某个阈值 (如 3 * MAD)，则判定为异常。

### 【2. 业务逻辑与状态流 (Business Logic & State Flow)】
*   **核心业务目标**: **测量一个 12V 铅酸电池或锂电池的 mΩ 级内阻，并通过级联网络上报。**
*   **全局状态机 (Global State Machine)**:
    *   `SM_IDLE`: 系统空闲，等待 `Periodic Check_Timer` (1分钟) 或 `Comm Command_Timer` (主机轮询)。
    *   `SM_MEASURING`: 正在执行内阻测量。此状态包含子状态机 (`Sub_SM_ACQ`, `Sub_SM_CALC`)。
    *   `SM_TEMP_MEASURE`: 正在执行 30s 周期的 DS18B20 温度测量。
    *   `SM_COMM_RX`: 正在接收 UART 报文。基于接收到的报文，可能跃迁至 `SM_COMM_PROCESS`。
    *   `SM_COMM_PROCESS`: 解析接收到的报文并生成响应或执行命令 (如：主机发送“读取数据”命令)。
    *   `SM_ERROR`: 检测到严重错误 (如：`IWDG` 复位、`HardFault`、`DS18B20` CRC 校验失败、ADC 采样溢出)。
*   **核心数据流向图**:
    `TIM3` -> `PWM (1kHz)` -> `模拟恒流源` -> `电池` [电流电压]
    `TIM3 (TRGO)` -> `ADC1 (10kHz DMA)` -> `ADCDMA_Buffer[256]` (电流与电压双通道)
    `Main Loop` -> `State Machine` -> `从 ADCDMA_Buffer 提取有效数据` -> `CMSIS-DSP 计算` -> `Float Impedance` -> `UART_TX_Buffer`

### 【4. 风险与 FMEA 分析 (Risks & FMEA)】
*   **FMEA 1：DS18B20 时序中断 (FMEA ID: F1)**。
    *   **失效模式**: `TIM14` 定时器中断被更高优先级的中断 (如 `USART1_IRQHandler` 或 `ADC1_IRQHandler`) 频繁打断，导致 OneWire 时序的微秒级延时出现偏差，通信失败。
    *   **影响**: 温度测量值错误 (可能停留在旧值)，或导致状态机陷入超时循环，阻塞其他任务。
    *   **缓解策略**: **中断优先级精心设计**：`USART1` > `TIM14` > `ADC1`。确保 DS18B20 的时序可以优先于 ADC 计算。**或者，使用硬件 SPI 模式模拟 OneWire。** 但 G0 系列 SPI 的极性与相位配置可能不完美。**更推荐使用 `TIM` 的输入捕获或输出比较功能来精确控制时序，避免纯软件延时。**
*   **FMEA 2：ADC 缓冲区溢出 (FMEA ID: F2)**。
    *   **失效模式**: `ADC1_IRQHandler` 中断处理时间过长 (如执行了浮点乘法)，导致下一次缓冲区更新到来时，上一次的数据尚未被读取，造成数据丢失或覆盖。
    *   **影响**: 内阻计算使用的数据不完整或包含新旧混合数据，导致计算结果严重偏差。
    *   **缓解策略**: **采用双缓冲 (Ping-Pong Buffer)**。使用 `DMA` 的 `CIRC` (循环) 模式，缓冲区大小设置为 16 个样本。`ADC1_IRQHandler` 中仅做 DMA 标志位切换和 `memcpy` 快速拷贝，不进行任何计算。计算全部放在主循环中，状态机负责检查半满/全满标志并启动计算。
*   **FMEA 3：UART 总线风暴 / 地址冲突 (FMEA ID: F3)**。
    *   **失效模式**: 在级联网络中，两个或多个节点认为自己是“主机”，同时向总线发送帧，或地址分配协议因干扰而失效。
    *   **影响**: 整个通讯网络瘫痪，所有节点进入错误状态，无法采集数据。
    *   **缓解策略**: **主从识别硬核化**：根据 `M/S` 引脚 (pin 12) 的上拉/下拉状态，在 `HAL_Init()` 后立即确定角色。**地址分配采用“请求-应答-确认”三向握手协议**。每个节点在收到的指令帧中提取目标地址，非目标地址节点保持沉默。

### 【6. 接口与集成契约 (Interfaces & Integration Contracts)】
*   **硬件物理接口**:
    *   `TIM3_CH3` (Pin 15): 方波，频率 1kHz ±0.01%，占空比 50%±1%，输出模式为 `TIM_OCMODE_PWM1`。 **应通过 RC 滤波电路后连至运放。**
    *   `VOLT_ADC` (Pin 8) / `CURRENT_ADC` (Pin 13): 阻抗匹配：ADC 输入阻抗需远大于前端运放输出阻抗。需确认是否需外部缓冲。ADC 采样率 10kHz，**必须由 TIM3 的 TRGO 边沿触发，确保相位同步。**
    *   `UART1_TX/RX` (Pin20/1): 电平标准 3.3V (LVTTL)，由 `CA-IS3721LS` 隔离后变为 5V。波特率 115200，8-N-1。必须使用 `DMA` 接收。
*   **软件内部接口**:
    *   **模块间 IPC**: 使用全局 `volatile` 结构体 `struct_msg_t` 传递简单的命令和数据。禁止使用复杂队列。例如，定时器中断服务程序 `TIM14_IRQHandler` 设置一个 `volatile uint8_t flag_timer_1s` 给主循环。
    *   **共享内存保护**: **必须保证 `DMA` 操作是原子性的**。当 `ADC1_IRQHandler` 正从 `DMA_Buffer` 读取数据时，不允许主循环访问。通过互斥锁 (如 `__disable_irq()` / `__enable_irq()`) 或利用 `DMA` 的半满/全满标志实现。
    *   **回调函数契约**: 外部 ISR (`stm32g0xx_it.c`) 调用的 `SCHED_IncTick()` 和 `UART_IRQHandler()` 已是虚函数桩。 `UART_IRQHandler` 接收一个 `uint32_t id`。必须确保实现中，`UART_IRQHandler(1)` 处理 UART1，`UART_IRQHandler(2)` 处理 UART2。
*   **外部通信协议**:
    *   **帧结构**： `0xAA` (帧头) + `1 Byte` (地址: 0=主机, 1-254=从机) + `1 Byte` (命令: e.g., 0x01 读取数据, 0x02 分配地址) + `N Bytes` (数据) + `2 Bytes` (CRC-16-IBM: MSB First, Poly=0x8005, Init=0x0000)。

### 【7. 深度测试与验证策略 (Deep Test & Validation Strategy)】
*   **SIL (Software-in-the-Loop)**:
    *   **目标**: 在 PC 上 (使用 Keil 仿真器或 VS Code + 插件) 验证核心算法，如 `arm_mult_q15` 的内阻计算流程、同步解调算法、MAD异常检测逻辑。
    *   **方法**: 使用 `__STATIC_INLINE` 宏和 `volatile` 模拟硬件寄存器。编写一个 Python 或 C 脚本生成已知内阻（如 50mΩ）的模拟 ADC 数据流 (带白噪声)。将其与 S-Function 或 C 测试桩结合，输入到算法模块，比较输出结果。 **误差应 < 1%**。
*   **HIL (Hardware-in-the-Loop)**:
    *   **目标**: 在实际 PCB 或开发板上验证所有外设时序和通信。
    *   **设备**: 信号发生器 (产生 1kHz 正弦波模拟电池电压)、示波器 (验证关键波形)、电子负载 (调整电池模拟器内阻)。
    *   **测试**: 示波器探头接 `TIM3_CH3` 和 `ADC1 触发信号`，确认相位关系。使用信号发生器注入一个已知幅值的 1kHz 交流信号到 ADC 输入端，验证测量值正确。
*   **时序/物理层验证**:
    *   **方法**: 使用逻辑分析仪 (8通道) 连接在 `TIM3_CH3`, `UART1_TX`, `UART1_RX`, `DQ` 上。
    *   **验证**:
        1.  1kHz PWM 占空比和频率精度。
        2.  主机向从机发送“分配地址”帧后，在 DQ 线上是否能观测到预期的 DS18B20 时序 (注意：DS18B20 是总线设备，与 UART 共用 MCU 时间)？**这是测试关键点**。
    *   **编译器优化测试**：分别使用 `-O0` 和 `-Os` 优化等级编译，对比 ISR 的执行时间 (通过示波器测量 GPIO 脉冲)。
*   **故障注入 (Fault Injection)**:
    *   **方法**: 手动或通过上位机模拟。
    *   **测试 1**: 断开 DS18B20 (DQ 引脚悬空)，观察状态机是否进入 `SM_ERROR`，并通过 LED (如外接LED闪烁) 或 UART 日志上报。
    *   **测试 2**: 通过上位机向 MCU 发送无帧头、校验错误的 UART 报文，观察接收状态机是否能够恢复 (超时后重置 `rx_index`)。
    *   **测试 3**: 使用可调电源模拟电池电压从 0V 快速上升到 16V，观察 ADC 采样值与异常检测逻辑是否工作。
*   **计量与校准 (Metrology)**:
    *   **流程**:
        1.  连接已知的**精密电阻箱** (如 0% 误差，范围 1mΩ ~ 1000mΩ) 到电流注入输出端和电压采样输入端。
        2.  在一定温度 (如 25°C) 下进行最***小2点校准 (如 10mΩ 和 100mΩ)。记录 `ADC采集值` 与 `真实内阻` 的关系，通过**线性拟合**计算校准系数。
        3.  在 0°C ~ 60°C 范围内重复步骤 2，**验证温漂**，并决定是否需要温度补偿。
        4.  重复测量 10 次，计算**重复性** (标准方差)。

### 【8. 性能、资源与规模 (Performance, Resources & Scale)】
*   **资源预算表**:
    *   **Flash**: 32KB 总量。估计：CMSIS-DSP (6KB)，HAL (8KB)，应用层 (4KB)，协议栈 (2KB)，启动/系统 (2KB) -> 剩余约 10KB。
    *   **RAM**: 8KB 总量。估计：栈 (Heap+Stack) 共 2KB，全局变量 (状态机、DMA缓冲、UART缓冲) 4KB，堆 (C库动态分配) 1KB -> 剩余约 1KB 用于其他临时变量。**必须严格控制栈使用**，对 DS18B20 或复杂算法进行递归调用 (不建议) 需谨慎。
    *   **关键地址**：`DMA_Buffer` 必须放在 `SRAM1` 的前 0x2000 字节内 (因为 DMA1 只能访问这部分地址)。查阅 RM0440 确认。
*   **WCET (最坏情况执行时间) 分析**:
    *   **最高频中断**: `ADC1_IRQHandler` (10kHz, 每 100us 触发一次)。
    *   **WCET**: 假设中断内需执行 10 次乘法 (CMSIS-DSP) 和一次快速拷贝 (memcpy)。预估 64MHz 下，约 50-100 个指令周期 (约 0.7-1.5us)。**远小于 100us 的周期**。但需考虑 `UCART1_IRQHandler` (高优先级) 的抢占。若 `UART1` 高速通信 (115200bps，每 86us 触发一次 ISR)，与 ADC 中断交织，可能导致 ADC WCET 超过 50us。**需确保 UART 中断服务程序极短——仅读取一个字节并存入缓冲，不下重功夫解析。**
*   **系统扩展瓶颈**:
    *   **瓶颈**: **UART 总线带宽**。在级联模式下，主机需要对所有从机进行轮询。假设主机每 1 秒轮询一次所有节点 (如 5 个)，每个节点应答 10 字节数据，总数据量为 5*10*8 = 400 位，在 115200bps 下，需时约 3.5ms。**不是瓶颈**。
    *   **真正瓶颈**: **MCU 处理能力**。当网络规模扩大 (如 50 个节点)，轮询周期会变长，主机需要更多时间处理协议，而每个从机也需要在被询问时才能上报数据，这会影响数据刷新率。

### 【9. 安全与鲁棒性 (Safety & Robustness)】
*   **物理安全**:
    *   **隔离**：CA-IS3721LS 隔离 UART 信号，VPS8701B 隔离电源。**必须保证逻辑/物理隔离间距**。
    *   **输入保护**：过流保护由 F1-1 (PTC) 提供。TVS管 D1 钳位总线上的电压尖峰。
    *   **防反接**：电源输入端（+12V）未明确设计反接保护。**建议在 +12V 输入端串联一个肖特基二极管或 PMOS 管。**
*   **软件鲁棒性**:
    *   **防数组越界**: 在 UART 接收状态机中，必须有 `if(rx_index >= RX_BUFFER_SIZE)` 的判断，若溢出则丢弃整个帧，重置状态。
    *   **喂狗策略**: 应在**主循环中固定位置**喂狗 (IWDG)。如果状态机死锁于某个子状态 (如 SM_MEASURING 中的死循环)，未被喂狗，导致系统复位。**不能在 ISR 中喂狗**。
    *   **异常数据剔除**: 使用 MAD 算法。计算窗口内 (N=7) 数据的中位数 `M` 和绝对偏差 `MAD = median(|xi - M|)`。若 `|x_new - M| > k * MAD` (k=3)，则丢弃。
*   **攻击面 (物联网场景)**:
    *   **防重放攻击**: 级联协议帧中加入时间戳或序列号。
    *   **固件防篡改**: 可以通过 CRC 校验固件映像。但限于 Flash 32KB，可忽略。

### 【10. 可维护性与演进 (Maintainability & Evolution)】
*   **模块化边界**:
    *   `Drv` 层: 直接操作寄存器或 HAL 的外设驱动 (如 `drv_adc.c`, `drv_tim.c`, `drv_ds18b20.c`)。
    *   `Mid` 层: 协议栈 (`mid_uart_protocol.c`)，状态机框架 (`mid_scheduler.c`)。
    *   `App` 层: 核心业务逻辑 (`app_battery_monitor.c`)。
*   **黑匣子机制**:
    *   **目标**: 将最近的异常事件记录到 Flash (如 64 字节)。
    *   **实现**: 使用 `stm32g0xx_hal_flash.h` 中的 `HAL_FLASH_Program(FLASH_TYPEPROGRAM_WORD, ...)`。将错误代码、系统时间戳、运行标志等放入一个 `struct`，在进入 `SM_MEASURING_FAIL` 或 `SM_COMM_CRC_ERROR` 时写入 Flash 最后 1KB 区域。
    *   **掉电保存**: 无需专用 EEPROM。G030F6 的 Flash 可以擦写 1000 次以上，对于记录错误日志足够。
*   **OTA 升级 & 现场校准**:
    *   **OTA**: 通过 UART 进入专用的“Bootloader”模式。需预留 Flash 中的 Bootloader 区域 (约 4KB)。**当前项目未明确要求 OTA，但需保留接口。**
    *   **现场校准参数**: 存储在 Flash 最后 1 个 Sector 中。通过 UART 发送特定命令 (`0x03`)，可以更新校准系数。

### 【11. 用户体验与反馈 (UX & Feedback Mechanisms)】
*   **无头系统反馈**:
    *   **状态指示**: 预留一个 GPIO 驱动 LED (可通过 N-MOSFET 驱动)。
        *   **LED 恒亮**: 系统启动正常。
        *   **LED 快闪 (10Hz)**: 正在执行内阻测量。
        *   **LED 两短一长闪**: 错误状态 (e.g., DS18B20 故障或 UART 通信超时)。可让 LED 闪烁特定的错误码。
    *   **蜂鸣器**: 预留 PWM 脚驱动蜂鸣器，用于报警或确认。
*   **生产/装配环节**:
    *   **一键自检**: 设计一个短按按键。按下后，系统进入 `SM_SELFTEST` 状态。自动完成：`TIM3` 波形输出检查 (通过检测反馈信号)，`ADC` 零位检查，`DS18B20` 通信检查，`UART` 回路检查。若全部通过，LED 长亮 3 秒。若有失败，通过 LED 闪码报告错误。

### 【12. 时间节奏与关键路径 (Timeline & Critical Path)】
*   **并行策略**:
    *   **硬件打板 (7-10 天) + 固件开发 (0 日)**: 立即使用 **NUCLEO-G031K8** 开发板开始固件开发。
    *   **核心逻辑开发 (2 天)**: 完成状态机框架、PWM/ADC/DMA/DS18B20 驱动、SIM 仿真算法。
    *   **核心算法调优 (1 天)**: 在开发板上挂一个示波器，使用已知电阻箱校准内阻算法。
*   **强耦合路径 (串行)**:
    1.  **底层驱动 (所有) -> 状态机框架**: 必须先完成所有外设（ADC, TIM, UART, GPIO, DS18B20）的初始化驱动，才能构建容器化的状态机。**这是关键路径的起点。**
    2.  **UART 协议栈 -> 主从/级联功能**: 必须先完成帧解析和 CRC 校验，才能实现地址分配和轮询。**这是第二个关键路径。**
    3.  **模拟前端 (HIL) -> 算法最终验证**: 必须收到完整且成品的 PCB，才能进行最终的内阻精度、抗干扰和温漂测试。**这是最后一个关键路径。**

## critical_risks
*(已在【4. 风险与 FMEA 分析】中详细展开)*

1.  **DS18B20 时序中断** (FMEA ID: F1) - 缓解：精心设计中断优先级，使用 `TIM` 输入捕获。
2.  **ADC 缓冲区溢出** (FMEA ID: F2) - 缓解：双缓冲 DMA + 主循环计算。
3.  **UART 地址冲突/总线风暴** (FMEA ID: F3) - 缓解：硬件引脚识别主从 + 三向握手协议。

## open_questions
1.  **U8, U9, U10, U11 的具体型号和功能是什么？** (推断是运放/复用器) – **必须确认**，缺失则无法设计正确的ADC前端匹配和驱动。
2.  **交流恒流电路的 RC 滤波器参数 (R, C 值) 是什么？** – 直接影响 `TIM3_CH3` 的频率设置和空占比设计，以及系统相位偏移。**必须索取**。
3.  **电池的连接方式是？ (2线法 vs 4线开尔文法)？** – 网表中 `VOLT_ADC` 和 `CURRENT_ADC` 是分开的。如果是4线法，测量精度会高很多。**必须明确**。
4.  **是否预留了外部按键用于自检或校准？** – BOM中未看到按键。如果不预留，自检模式只能通过 UART 命令触发，但需要外部上位机参与，降低了现场易用性。
5.  **级联网络的最大节点数是多少？** – 影响地址分配策略和轮询周期。建议设计为支持最多 32 个节点。
