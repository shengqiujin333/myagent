# 项目战略与方向性思考地图 (Thinking Map)

## Project Vision & Boundaries

- **项目愿景**: 构建一个基于STM32G030F6P6的分布式、级联式、高鲁棒性的电池内阻在线测量系统。该系统应能在复杂的电池充放电环境中，提供准确、可靠且可追溯的内阻数据，并具备强大的异常诊断和现场可维护性。
- **硬性边界**:
    - **计算与存储边界**: 64KB Flash / 8KB RAM的物理天花板。任何软件开发都必须在成本允许的范围内优化，必须对“暗物质”开销（如HAL库、DSP库）持零容忍态度。
    - **物理接口边界**: TSSOP-20有限的GPIO引脚数（可用约14个）和已知外设(ADC/TIM/UART)。这决定了所有功能的实现必须基于这些有限的硬件资源，无扩展可能。
    - **系统级联拓扑**: 物理层为“手拉手”菊花链拓扑。所有核心业务逻辑（状态机、协议、异常处理）都必须以此为核心约束进行设计。
    - **非阻塞铁律**: 系统所有时间关键路径（1kHz PWM同步、10kHz ADC采样、50Hz半周期通信）绝对禁止任何形式的阻塞，包括但不限于`HAL_Delay`、轮询等待、`while`循环。
    - **依赖不确定性边界**: UART1在TSSOP-20上的具体引脚分配、隔离侧模拟电路的精确拓扑、以及运放的具体型号未知。软件架构必须设计出抽象层以隔离这些不确定性，允许硬件未就绪时进行Mock开发和测试。

## 12-Dimension Directional Sweep

### 1. Tech Stack & Critical Path
- **方向指导**:
    - **MCU & 工具链**: STM32G030F6P6 + Keil MDK V6.19是既定事实。必须深入评估 HAL 库的模块级裁剪可能性。对于ADC和TIM的配置，考虑如果HAL函数导致不可接受的时序抖动（如在中断中），则应降级为使用LL库直接操作寄存器。
    - **关键路径识别**:
        1.  **ADC采样闭环**: `TIM1_TRGO (10kHz更新事件) -> ADC注入组采样 -> DMA传输完成中断 -> 中断内数据预处理（滑动平均/相位校正）`。这是系统最高频（10kHz）且最关键的路径。
        2.  **1kHz PWM输出**: `TIM3 -> CH3 (PA7) -> RC滤波 -> ...`。此路径的相位抖动会直接影响测量的同步性。
        3.  **UART1主机轮询**: `主机状态机 -> 发送查询指令 -> 等待从机回复超时/成功 -> 解析数据`。此路径决定了整个级联网络的吞吐量和响应速率。
    - **潜在瓶颈方向**:
        - **10kHz中断入口**: 如果单个ADC采样DMA中断处理时间超过100us，将会导致下一个采样点丢失。必须评估在此中断内执行的任务量（如简单的累加和标志设置，而非复杂FFT）。
        - **UART接收中断**: 在大数据量（如多个从机上报）时，UART1中断可能处理不过来，导致溢出。必须评估DMA+IDLE中断或专用FIFO缓冲的方案。

### 2. Business Logic & State Machine
- **方向指导**:
    - **核心全局状态机**: 应划分为三个并行协作的有限状态机。
        1.  **通信状态机**: IDLE -> ADDR_ASSIGN -> POLL_MASTER -> POLL_SLAVE1 -> ... -> POLL_SLAVEN -> POLL_MASTER。
        2.  **测量状态机**: IDLE -> ADC_ACQ (10kHz连续) -> DFT_FILTER -> MEAS_CALC -> ANOMALY_CHECK -> DATA_READY。
        3.  **温度状态机**: IDLE -> DS18B20_CONVERT -> DS18B20_READ -> TEMP_READY。
    - **异常与降级策略**:
        - **通信中断**: 如果主机连续N次向某一从机轮询失败，应标记该节点为“离线”，继续轮询其他节点，并产生故障事件。不要阻塞整个链路。
        - **传感器失效**: 如果CURRENT_ADC或RES_ADC的ADC值固定在供电轨或GND，应立即判定为硬件故障，停止内阻计算，并输出错误码。
        - **DS18B20故障**: 如果连续两次读取失败（CRC错误或无响应），应使用预设的默认温度值（如25°C）进行内阻计算，并标记温度传感器故障。
    - **看门狗机制**: 状态机调度器必须有一个**主循环看门狗**。每次主循环遍历完所有状态机后，喂一次独立看门狗（IWDG）。如果某个状态机死锁，主循环卡死，IWDG复位系统。

### 4. FMEA Risk Direction & Mitigation
- **致命失效模式方向**:
    1.  **方向1: ADC同步采集失锁**: 由于TIM1配置错误或时钟树混乱，导致10kHz ADC采样与1kHz PWM相位失锁，测量结果完全错误。
        - **缓解**: 启动时执行同步校准：测量一个已知的低噪声电阻（如0.1%精密电阻），若DFT后基波幅值偏差超过阈值，则报警并修正采样触发延迟。
    2.  **方向2: UART级联链路死锁**: 某个从机UART因电源纹波或静电干扰导致状态机进入未定义状态，持续发送垃圾数据或完全不回复，导致主机轮询超时挂起。
        - **缓解**: 主机侧必须设计极短的超时（如100ms）和重试机制。从机UART的接收状态机必须设计“字符间超时”看门狗，若超过10ms无正常数据，自动复位接收状态机。
    3.  **方向3: 校准系数丢失或损坏**: 存储在Flash中的多点校准系数因电源跌落或写操作中断而损坏，导致内阻值产生巨大系统误差。
        - **缓解**: 采用双备份存储策略。写入时先写备份区，再写主区 → 读取时先读主区，校验和错误则自动从备份区恢复并重新计算。同时，每一次写入前检查Flash处于稳定电压。

### 5. Dependencies & Boundaries
- **硬性边界**:
    - **强依赖硬件/IP**: TSSOP-20封装下的GPIO复用AF表（具体到UART1_TX/RX的引脚号）、模拟RC滤波器的截止频率与元件容差、隔离电源的启动时间。
    - **外部物理约束**: 电池内阻测量的电磁干扰环境（充电机的大电流充放电、BMS的DC-DC噪声）、工作温度范围（-40°C ~ 85°C）、DS18B18单总线对线缆长度的敏感度。
- **软硬件解耦方法论**:
    - 对**所有硬件外设**（UART, TIM, ADC, GPIO）的初始化函数，将其抽象为 `SYSTEM_HW_Init()` 接口，内部调用HAL/LL库。在软件测试时，提供一个空实现或模拟实现的`SYSTEM_HW_Mock_Init()`。
    - **算法模块**（内阻计算、异常检测、相位校正）应完全与硬件无关，只依赖标准C数据类型，便于在PC上进行SIL测试。

### 6. Interfaces & Integration
- **接口契约原则**:
    - **内部模块间**: 使用 `状态机事件队列 + 全局状态结构体` 进行通信。事件队列使用循环缓冲区实现，禁止在中断中直接修改全局状态。
    - **软硬件间 (BSP层)**: BSP函数（如 `BSP_ADC_GetValue()`）必须返回成功/失败状态码，并保证非阻塞。
    - **外部通信 (UART协议)**: 帧格式必须包含**起始符、设备地址、帧ID、数据长度、数据负载、CRC16校验、结束符**。所有多字节数据采用小端序。必须设计防截断机制：接收方必须在接收过程中持续检查超时，一旦超时，立即丢弃当前不完整帧并复位接收状态机。

### 7. Deep Test & Validation Strategy Matrix
- **SIL/MIL (软件在环)**:
    - **测试目标**: 内阻计算、异常检测、多点校准算法。
    - **测试方法**: 创建一个独立的PC端C工程（如VS Code + GCC/MinGW），调用相同的算法函数。生成已知的正弦波+白噪声的ADC原始数据文件，作为测试输入。自动化断言输出结果是否在理论偏差范围内。
- **HIL (硬件在环)**:
    - **测试目标**: 验证系统在极限物理条件下的响应。
    - **所需台架方向**: 精密可编程直流电源（模拟电池端电压变化）、高精度电子负载（模拟充放电电流）、高精度信号发生器（模拟干扰信号注入）、屏蔽箱（模拟强EMI环境）。
- **时序与物理层验证**:
    - **仪器**: 至少4通道示波器（≥200MHz带宽，带协议解码功能）和逻辑分析仪。
    - **验证点**: 同步验证TIM3_CH3的PWM输出与ADC的采样触发信号；测量UART1总线上的帧时序及电平质量。
- **故障注入**:
    - **方法**: 使用逻辑分析仪或专门的故障注入板，对UART总线注入单比特翻转、帧断裂、间隙毛刺。对电源域进行电压跌落测试。对ADC输入信号注入随机噪声。
- **计量与校准**:
    - **环境**: 恒温箱（控制校准时的环境温度）。
    - **流程**: 使用0.1%或更高精度的标准电阻箱，对每个量程进行多点标定。记录校准数据并存储在Flash中。校准报告应作为固件发布的一部分。

### 8. Resource & Performance Evaluation
- **优化方向**:
    - **HAL库裁剪**: 编译后使用IDE生成的.map文件，详细分析哪些HAL模块和函数被实际链接。移除所有未使用的HAL/CMSIS文件。必要时，手动编写最精简的LL函数。
    - **RAM优化**:
        - 所有全局的中间计算缓冲区，如`sine_f32[256]`，必须使用`const`限定符放入Flash。
        - 通信缓冲区（如UART接收FIFO）的大小必须精确计算，避免浪费。
    - **CPU性能**:
        - 10kHz ADC中断内只做**数据累加和标志置位**，将复杂的DFT/同步平均计算放在主循环中处理。
- **评估方法**:
    - **Stak峰值监控**: 在启动的调试阶段，在每个任务/中断的入口和出口填充特定的字节模式（如`0xDEADBEEE`），运行一段时间后，通过调试器检查内存，找出未被覆盖的最大栈深度。
    - **WCET估算**: 对10kHz ADC中断和UART中断，使用`DWT_CYCCNT`寄存器进行实测计时，确保其在所有可能路径下均小于100us。

### 9. Safety & Defensive Programming
- **物理安全**:
    - **保护方向**: 输入信号（VOLT_ADC, RES_ADC）应有过压保护钳位电路，采样电阻有过流保护。DS18B20单总线应具备静电放电保护。
- **信息安全**:
    - 作为一个无智能系统的嵌入式设备，信息安全威胁较低。但仍应防范：UART指令中包含恶意命令（如非法写入Flash）。协议设计应包含白名单机制，只响应已知的合法命令。
- **防御性编程规范**:
    - **所有外部输入视为不可信**: 解析UART帧时，必须校验CRC16、帧长度、数据内容合理性（如内阻值>1M欧姆视为无效）。
    - **指针防护**: 所有函数指针调用前必须检查是否为NULL。
    - **数组越界**: 循环数组索引必须增加边界检查，使用 `if (index >= ARRAY_SIZE) return ERROR;`。
    - **变量初始化**: 所有局部变量和全局变量在声明时立即初始化。

### 10. Maintainability & Observability
- **可观测性**:
    - **无头系统**: 主板上至少需要两个LED：一个绿色（系统运行正常指示灯，由主任务周期性翻转）、一个红色（故障指示灯，常亮表示严重故障，不同闪烁频率编码不同错误类型）。
    - **黑匣子 (Blackbox)**:
        - 将最后N次（如100次）测量结果（电压、电流、内阻、温度）和一个环形缓冲区的故障事件日志（时间戳+错误码+关键变量）实时保存到内部Flash的末尾区域。
        - 当系统复位后，主机可以通过特殊指令读取这些数据，用于现场故障分析。
- **可维护性**:
    - 代码书写遵循统一的编码规范（如MISRA C 2004的子集），所有模块必须有清晰的DOXYGEN风格注释。

### 11. UX & Production Test
- **产线测试**:
    - **一键自检**: 硬件上电后，固件应首先执行自检流程：检查ADC基准电压是否正常、所有温度传感器是否通信正常、用于内阻计算的ADC通道是否与标准电阻箱连接。
    - **自动化产线校准**: 产线流程应为：ATE测试系统通过UART发送“进入校准模式”指令 → 固件自动切换到校准状态机 → ATE依次切换已知的N个标准电阻 → 固件采集并记录ADC读数 → ATE发送“保存校准系数”指令。
- **用户体验**:
    - 当出现异常数据时，UART输出的数据包应包含 `DATA_FLAG` 字段，用于指示该数据是正常/警告/故障，便于上位机进行分析。

### 12. Timeline & Parallelism
- **长周期物料/任务**:
    - **硬件设计与打板**: PCB Layout、BOM确认、焊接、初样调试。
    - **隔离侧模拟电路调试**: 恒流源、RC滤波器的性能验证和调优。
- **并行策略**:
    - **第一周**: 硬件团队提供初步原理图和网表；软件团队基于此启动**核心算法SIL开发**（内阻计算、异常检测）。**UART协议设计**也在此阶段完成并文档化。
    - **第二周**: 硬件团队交付PCBA样板；软件团队完成**所有可Mock的模块**（DS18B20、UART驱动、ADC采集状态机）的开发和单元测试。在x86 PC上搭建虚拟目标进行测试。
    - **第三周**: 软硬件联调。重点是**10kHz ADC同步采集**和**UART级联通讯**的调试。使用逻辑分析仪验证协议。
    - **第四周及以后**: 系统集成测试、压力测试、EMC预测试、产生现场。

| **UART级联死锁** | 某从机MCU因静电或电源闪烁导致程序跑飞，其UART持续发送无意义的0x00或0xFF。 | 主机UART接收FIFO溢出，无法接收其他从机回复，导致系统通信瘫痪。 | 1. 主机侧增加字节超时看门狗，超过规定时间无正常数据，主动丢弃并重置UART接收状态机。<br>2. 从机侧实现IWDG，程序跑飞后1秒内自动复位。 |
| **校准系数损坏** | 写入Flash时遭遇电源跌落，导致分区部分写入或全零。 | 内阻计算值产生恒定的系统偏差，无法挽回。 | 1. 双备份存储+校验和。<br>2. 每次上电读取时自动校验并从备份恢复。<br>3. 在电压监控电路发出掉电警告后，立即停止正在进行的Flash写操作。 |
| **DS18B20总线锁死** | DS18B20在强EMI干扰下，产生总线冲突，将所有从机输出拉低。 | 温度测量功能失效，且可能影响其他IO。 | 1. 使用开漏输出并利用外部4.7k上拉电阻。<br>2. 在时序的“写时隙”和“读时隙”之间增加严格的延时控制，避免MCU进入死锁。<br>3. 主控GPIO配置为开漏模式，可被从机强行拉低，但通过内部空闲周期检测来恢复。 |
