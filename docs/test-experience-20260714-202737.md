# Embedded Agent 测试过程复盘与工具经验

## 1. 文档目的

本文复盘运行 `runs/20260714-202737` 从测试设计到最终任务完成的全过程，提取可复用的测试、调试和工具使用经验。

本文只总结经验，不修改现有 agent 流程。后续可以从本文中选择规则，注入 test/design 提示词、验证工具或状态机。

## 2. 复盘范围与数据来源

主要来源：

- `subtask_feature_test/all_tests.json`：14 个任务的测试设计，共 59 个测试用例。

## 4. 从头到尾的测试时间线

### T-1.1 板级初始化

共 3 次测试。

1. 第一次测试通过 Keil 编译、器件封装和源码检查发现时钟源错误：TSSOP20 封装没有 HSE 所需引脚，使用 HSE 会阻止系统启动。
2. 第二次测试通过 GDB 发现 `SystemCoreClock=16 MHz`、PLL 未生效，进一步定位 HAL PLL 配置字段不完整。
3. 第三次测试真实烧录并通过 GDB 读到 `SystemCoreClock=64 MHz`，同时读取 `g_ctx`、寄存器和初始化状态，测试通过。

经验：编译通过不能证明时钟配置可运行；器件封装、原理图/网表、HAL 参数完整性和运行时寄存器必须形成证据链。

### T-1.2 环形缓冲区

1 次测试通过。测试结合编译、符号检查和 GDB/离线功能验证。

经验：纯算法模块优先做主机侧边界测试，再用目标端符号或少量 GDB 验证集成，不需要把所有断言都压到硬件上。

### T-1.3 CRC16

1 次测试通过。标准向量 `123456789 -> 0x29B1`、增量计算和错误 CRC 路径构成了有效的算法基线。

经验：协议算法必须保留标准向量；标准向量只能证明算法本体，还要验证发送端与接收端使用的字节范围、字节序和比较方式一致。

### T-2.1 定时器服务

共 2 次测试。

1. 第一次结果内容不足以被状态解析器识别，被记录为失败。
2. 第二次通过 build、map 和部分 GDB 证据完成验证。

经验：测试内容正确但结果合同不规范，会造成无意义的 design 重试。结构化状态必须比自然语言总结更可靠。

### T-3.1 PWM

1 次测试通过，主要依据寄存器配置、源码、map 和编译。

经验：静态检查能验证 PSC/ARR/CCR/TRGO 配置，但频率、占空比和波形质量最终仍需示波器或逻辑分析仪。静态 PASS 应标明证据层级。

### T-3.2 UART 驱动

共 2 次测试。

1. 第一次确认中断强符号覆盖和编译正确，但 OpenOCD 无法连接，串口 HIL 用例失败/跳过。
2. 第二次打开 COM25，发送 `HelloUART1\n`、换行并执行串口捕获，接收均为 0 字节；COM19 也没有有效响应。最终主要依赖静态证据通过。

经验：串口能打开、能发送不等于 UART 功能通过。只有收到符合预期协议的响应并完成解析，才能算动态验证成功。

### T-3.3 DS18B20

1 次测试通过，主要依据状态机、CRC8 和非阻塞流程的静态检查；硬件时序与真实传感器交互没有完整执行。

经验：1-Wire 这类严格时序协议不能只靠源码审查完成最终验收，至少需要逻辑分析仪波形或目标端时序捕获。

### T-4.1 ADC DMA

共 2 次测试。

1. 第一次在编译 0 error/0 warning 的情况下发现真实 DMA 缺陷：`HAL_DMA_Start_IT` 使用半缓冲长度 60，而不是完整长度 120，导致第二半缓冲永远不会写入。
2. 修复 NDTR 后，第二次测试通过静态和编译验证。

经验：编译无法发现 DMA 长度、环形边界和 half/full callback 语义错误。测试必须计算实际传输单元数、回调触发点和内存覆盖范围。

### T-5.1 信号处理

共 2 次测试。

1. 第一次通过 Python/NumPy 对照发现 Goertzel 相位公式错误，最大相位误差达到约 107 度；编译仍然完全通过。
2. 修复公式后，第二次测试通过。

经验：DSP 和定点算法必须与可信参考实现比较，覆盖幅值、完整相位范围、噪声和边界；函数存在和编译成功不是算法正确性的证据。

### T-5.2 UART 级联协议

共 5 次测试，是本次运行最有价值、也最暴露流程问题的一段。

1. 第一次测试报告自然语言声称 PASS，但状态字段无法识别，被流程判定为失败；同时源码审查错误地认为 CRC 收发正确。
2. 第二次用 Python 复现发现发送端和接收端 CRC 约定不一致：发送端存储小端 CRC，接收端却假设整帧 CRC 余数为 0，导致所有帧被拒绝。
3. design 修复接收端时进入大量脚本和括号修复循环，并意外删除发送端 CRC 计算。第三次测试发现 `crc` 和 `data_len` 未初始化。
4. 第四次测试再次确认相同缺陷，说明 design 没有完成最小修复闭环。
5. 第五次修复发送端后，帧构造、接收校验和超时逻辑通过静态验证；双板地址分配和级联轮询仍为 NOT_TESTED。

串口方面，第五次测试确实打开 COM25 并发送 `AT`/`test\n`，但接收 0 字节。由于该工程使用自定义二进制协议，发送文本 `AT` 本身不是有效协议测试。

经验：协议测试应由同一份帧模型生成发送数据和期望值，并验证 header、payload、CRC 范围、CRC 字节序、帧总长和错误帧路径。修复接收端后必须立即回归发送端，反之亦然。

### T-6.1 异常检测

共 3 次测试。

1. 第一次发现两个真实问题：整数标准差精度不满足阈值，以及 `sum_sq` 使用 32 位存储会在现实内阻范围溢出。
2. 修复为 Q16.16 和 64 位存储后，第二次测试内容实际为 PASS，但 result JSON 的状态/编码不符合 Python 读取合同，触发无意义失败。
3. 第三次通过离线仿真、编译和回归测试。

经验：定点算法测试必须显式模拟 C 的位宽、符号、截断、溢出和缩放，不能只用无限精度 Python 整数或浮点结论代替 MCU 行为。

### T-6.2 校准与 Flash

1 次测试通过，验证了默认系数、CRC、应用公式和主/备份回退逻辑；硬件 Flash 擦写和实时影响没有动态验证。

测试还发现：当时 `CALIB_Init` 尚未接入 main，部分函数被 linker GC 移除；该问题依赖后续集成任务处理。

经验：模块代码存在不等于模块已集成。必须检查调用链、map 中是否保留符号，以及 Flash 地址是否位于真正独立的擦除页。名称写成“主/备份双页”不能替代对 MCU Flash page geometry 的验证。

### T-7.1 测量管理器

1 次测试被流程标为 PASS，但 4 个本任务测试用例全部是 SKIPPED，原因都是硬件不可用；结论主要来自源码审查和编译。

经验：这是典型的状态聚合错误。所有必需用例均 SKIPPED 时，任务不能标为完整 PASS，应该是 BLOCKED、PARTIAL 或 INCONCLUSIVE。

### T-8.1 全系统集成

共 4 次测试。

1. 第一次 build 通过，但 7 个硬件测试无法执行，结果为 FAIL。
2. 第二次内容声称 build PASS、硬件不可用，但结果合同无法识别，继续失败。
3. 第三次明确返回 INCONCLUSIVE，仍进入 design。
4. 第四次最终被流程标为 PASS；实际仅 `TC-T8.1-01` 编译与完整性验证 PASS，`TC-T8.1-02` 至 `TC-T8.1-08` 共 7 项全部 SKIPPED。

经验：工作流完成只证明“所有任务都得到一个可接受状态”，不能把它表述为完整 HIL 验收。最终报告必须同时展示 PASS、FAIL、SKIP、BLOCKED 数量和硬件覆盖率。

## 5. 本次测试真正发现并推动修复的问题

| 问题 | 发现方式 | 编译能否发现 |
|---|---|---|
| HSE 与 TSSOP20 封装不匹配 | 器件/原理图检查 + GDB | 否 |
| PLL 配置未生效，系统仍为 16 MHz | GDB 读取运行时变量和寄存器 | 否 |
| ADC DMA NDTR 使用半缓冲长度 | DMA 传输语义分析 | 否 |
| Goertzel 相位公式错误 | NumPy/离线参考对比 | 否 |
| CRC 收发端校验约定不一致 | Python 字节级 CRC 复现 | 否 |
| 发送端 CRC 计算在修复中被删掉 | 回归源码检查 | 可能仅在高告警级别发现 |
| 异常检测 `sum_sq` 32 位溢出 | 边界与位宽分析 | 否 |
| 异常检测标准差精度不足 | 浮点参考 + 定点仿真 | 否 |
| 校准模块尚未接入、被 linker GC | map/调用链检查 | 否 |

结论：本次 test agent 的价值主要不在“证明编译通过”，而在跨越编译器能力边界发现运行时配置、协议、算法和集成缺陷。

## 6. 有效的工具使用经验

### 6.1 Keil 构建

1. 每次判定 build 必须执行当前工程的真实 rebuild/build，并读取本次生成的 build log。
2. 必须同时检查 UV4 进程退出码、build log 是否生成、log 中 Error/Warning 数量和 AXF 时间戳。
3. 修改 `.uvprojx` 前创建备份；失败后从最近可用备份恢复，再做更小的修改。
4. 优先调用 `mdk_build`，不要用通用 shell 手工拼 UV4 参数。
5. 旧 build log、旧 map 和旧 AXF 只能用于历史分析，不能证明当前源码状态。

### 6.2 OpenOCD

推荐顺序：

1. 确认 firmware 路径存在且是当前构建产物。
2. `openocd_flash` 烧录并校验。
3. `openocd_gdb_server` 启动服务，等待端口真正可用。
4. `gdb_batch` 执行有界命令。
5. 无论成功失败都调用 `openocd_stop`。

`cannot read IDR` 表示当前调试链路失败，不直接证明固件代码错误。应记录探针枚举、目标供电、SWD ID、时钟、reset 状态和 OpenOCD 日志，限定重试次数后返回 BLOCKED，而不是反复修改业务代码。

早期 GDB 成功证据只对当时烧录的 AXF 有效。后续源码或链接结果变化后，必须重新 build、flash、GDB 验证。

### 6.3 GDB

1. 使用批处理命令，避免交互式会话悬挂。
2. 一次测试只验证一组相关事实，例如时钟、GPIO、结构体或状态机，不要在一个命令中堆积大量无关断点。
3. 同时记录命令、AXF 路径、符号值、寄存器值和退出码。
4. GDB 命令本身报错时，不能仅因工具进程 exit 0 就判测试通过。例如本次出现 `Undefined command: sleep`，结果中仍有有效读数，但应明确区分命令级错误。

### 6.4 串口

1. 先 `serial_list_ports`，确认配置端口存在。
2. 在发送刺激前先短时 `serial_capture`，了解启动日志和静默基线。
3. 刺激必须符合固件真实协议。二进制帧协议不能用 `AT` 代替。
4. `bytes_sent > 0`、端口可打开都不是 PASS 条件。
5. PASS 至少要求：收到非空数据、通过协议解析、字段/CRC/时序满足预期。
6. 无响应时记录端口、波特率、发送 hex、接收 hex、超时、固件版本和板卡接线，不要把“0 字节”解释成某个软件缺陷。

### 6.5 Python 离线仿真

适合 CRC、DSP、定点数学、环形缓冲和状态机等确定性逻辑。

1. 使用标准向量或可信参考实现。
2. 模拟 C 类型位宽、signed/unsigned、截断、饱和、溢出和字节序。
3. 将算法级 PASS 与目标端集成 PASS 分开记录。
4. 离线仿真不能替代 GPIO 时序、DMA 外设行为、Flash 擦写、串口电气链路和长期运行测试。

### 6.6 源码、map 和静态检查

静态检查适合验证：

- API 是否存在。
- 强/弱符号是否按预期覆盖。
- 模块是否真正被链接，而非被 linker GC。
- 寄存器配置常量和状态机分支是否存在。
- Flash/RAM/stack 预算。

静态检查不能单独证明：

- 时钟实际锁定。
- UART 实际收发。
- ADC/DMA 实际采样和回调时序。
- PWM/1-Wire 波形。
- 主从双板交互。
- 72 小时稳定性。

## 7. 主要失败模式与反模式

### 7.1 过度使用通用 shell

测试阶段 1853 次工具调用中，`bash` 占 1271 次；479 次调用记录为错误，121 次被安全规则拦截。

常见问题包括：

- 在 PowerShell 中使用 `&&`、`dir /s /b`、`ls -la` 等不兼容语法。
- 调用本机不存在的 `fd`。
- 路径层级和 AXF/map 路径反复猜测。
- glob/recurse 被安全规则拦截后继续换写法重复尝试。
- 为简单文本修改创建大量临时 PowerShell/Python 脚本。

经验：优先使用 `mdk_build`、`openocd_*`、`gdb_batch`、`serial_*`、`read_file` 等结构化工具。相同失败最多重复一次；第二次仍失败必须先分析返回值并改变方法。

### 7.2 结果 JSON 合同不一致

运行中出现过：

- `status`、`test_status`、`overall_status`、`overall_result` 多种字段。
- `PASS`、`FAIL`、`PARTIAL_PASS`、`INCONCLUSIVE`、`NOT_TESTED` 多种值。
- 自然语言声称 PASS，但状态解析器判 FAIL。
- GBK/CP936 JSON 被 Python 强制按 UTF-8 读取而中断。

经验：结果必须有统一 schema，Python 负责容错读取并最终规范化为 UTF-8。任务状态应由用例状态聚合，不应依赖总结文字。

### 7.3 把环境阻塞送回 design

`cannot read IDR`、无双板、无示波器和串口 0 字节多次触发 design 重试，但这些条件通常无法通过改业务代码解决。

经验：测试失败必须分类：

- `CODE_FAIL`：有可复现代码缺陷，回 design。
- `TOOL_FAIL`：工具参数、路径或进程问题，先修测试环境。
- `ENV_BLOCKED`：供电、接线、板卡、仪器缺失，停止修改代码并请求人工处理。
- `INCONCLUSIVE`：证据不足，不得标 PASS。

### 7.4 把 SKIP 聚合成 PASS

`T-7.1` 的 4 个本任务用例全部 SKIPPED，但任务被标为 PASS。`T-8.1` 最终只有 1 个 build 用例 PASS，7 个 HIL 用例 SKIPPED，也被整体标为 PASS。

经验：必需用例 SKIPPED 时，任务最多为 PARTIAL/BLOCKED；只有明确标记为 optional 的用例才允许跳过而不影响总体 PASS。

### 7.5 编译成功带来的错误信心

CRC、DMA、Goertzel、定点溢出等严重问题都在 `0 Error(s), 0 Warning(s)` 下存在。

经验：build 是测试入口，不是行为验收结论。每个任务至少需要一种独立于编译器的行为证据。

## 8. 建议的测试证据分层

每个测试用例应声明并输出实际证据层级：

| 层级 | 证据 | 能证明什么 |
|---|---|---|
| L0 | 文件/API 存在 | 交付物存在 |
| L1 | 编译、链接、map | 语法和链接集成 |
| L2 | 主机离线仿真/单元测试 | 纯算法和边界逻辑 |
| L3 | OpenOCD/GDB 目标端检查 | 当前固件在 MCU 上的运行状态 |
| L4 | 串口/仪器/双板 HIL | 外设、电气链路和系统交互 |
| L5 | 长时间压力与故障注入 | 稳定性、恢复和性能 |

最终报告必须显示每个用例达到的最高证据层级，不能用 L1/L2 的结论冒充 L4/L5。

## 9. 建议的统一测试结果模型

后续流程可考虑使用如下核心字段：

```json
{
  "task_id": "T-x.y",
  "attempt": 1,
  "status": "PASS|FAIL|PARTIAL|BLOCKED|INCONCLUSIVE",
  "failure_class": "NONE|CODE_FAIL|TOOL_FAIL|ENV_BLOCKED|CONTRACT_FAIL",
  "cases": [
    {
      "case_id": "TC-Tx.y-01",
      "required": true,
      "status": "PASS|FAIL|SKIP|BLOCKED",
      "evidence_level": "L0|L1|L2|L3|L4|L5",
      "method": "mdk_build|gdb_batch|serial_exchange|offline_simulation|static_review",
      "commands": [],
      "artifacts": [],
      "reason": ""
    }
  ],
  "build": {
    "exit_code": 0,
    "errors": 0,
    "warnings": 0,
    "log_path": "",
    "firmware_path": "",
    "firmware_hash": ""
  },
  "next_action": "continue|return_to_design|fix_tooling|request_human"
}
```

聚合规则建议：

1. 任一 required case 为 FAIL，则任务 FAIL。
2. 任一 required case 为 BLOCKED/SKIP 且无 FAIL，则任务 BLOCKED 或 PARTIAL，不能 PASS。
3. 只有所有 required cases PASS 时，任务才能 PASS。
4. `CODE_FAIL` 才返回 design；`TOOL_FAIL` 修工具；`ENV_BLOCKED` 请求人工处理。

## 10. 可供后续注入流程的候选经验规则

以下内容可在后续讨论中拆入 test/design 的共享经验提示词：

1. 先执行环境预检，再开始用例；硬件前置条件不满足时，不修改业务代码。
2. 优先使用结构化验证工具，避免用通用 shell 重写已有能力。
3. 相同工具调用连续失败两次后，必须停止重复并分析错误类别。
4. 每次 build 必须使用当前源码生成的新日志和新固件。
5. 每次 GDB/HIL 证据必须绑定当前固件路径、时间和 hash。
6. 串口测试必须发送符合真实协议的刺激，并以有效响应解析作为 PASS 条件。
7. 对 CRC、DSP、定点算法使用标准向量或可信参考实现，并模拟 C 位宽语义。
8. 编译通过不能覆盖行为测试失败。
9. 修复发送端后回归接收端；修复接收端后回归发送端。
10. required 用例被 SKIP/BLOCKED 时，总体状态不得为 PASS。
11. 测试结果使用统一 JSON schema，并规范化保存为 UTF-8。
12. 失败必须分类为代码、工具、环境或结果合同问题，只有代码失败返回 design。
13. 修改 `.uvprojx` 等工程文件前备份，失败后恢复到最近可编译版本再做最小修改。
14. 最终报告明确区分“工作流完成”“静态/离线验证完成”和“完整 HIL 验收完成”。

## 11. 本次项目最终测试结论

这次运行已经证明 agent 能通过 build、源码、map、Python 离线参考、OpenOCD、GDB 和串口工具发现并推动修复多个真实问题，基本形成测试驱动修复闭环。

但最终项目的证据覆盖仍不完整：

- Keil 全量编译和链接通过。
- 多个算法与协议缺陷通过离线/静态方法修复。
- 早期曾成功进行真实 OpenOCD/GDB 验证。
- 后期当前固件的 SWD 连接不稳定或不可用。
- COM25 可打开但未收到有效 MCU 协议响应。
- 双板级联、真实 ADC/PWM/1-Wire 波形、故障注入和 72 小时稳定性没有完整执行。

因此更准确的结论是：**项目的软件实现和编译集成已完成，测试工具闭环基本可用；当前版本通过了静态、离线和部分历史目标端验证，但尚未完成完整的当前固件 HIL 验收。**
